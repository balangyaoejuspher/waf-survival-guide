# OWASP CRS Rule `920350` â€” Host Header is a Numeric IP Address

> **Scope of this file:** the single canonical reference for OWASP CRS rule `920350` â€” protocol-enforcement rule that denies any request whose `Host` header is a numeric IPv4 or IPv6 address (instead of a hostname) â€” and the false positives it produces on internal LB health checks, container-to-container service mesh probes, Kubernetes liveness/readiness checks hitting Pod IPs, raw-IP development probes, and scanning tools that pin requests by IP. Covers Google Cloud Armor, AWS WAF, and Cloudflare WAF side by side.
>
> Companion to [docs/rules/920280.md](920280.md) (missing Host). The two often co-cluster in the same triage.

---

## 1. The Symptom

| Cross-provider signature                                                                                                                                   | Where you see it        |
| ---------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------- |
| `HTTP 403` on requests that target a raw IPv4 / IPv6 address in the `Host` header (e.g. `Host: 10.0.0.42`, `Host: 203.0.113.5:443`, `Host: [2001:db8::1]`) | Edge log                |
| Internal Kubernetes liveness probe hitting `Host: <pod-ip>` returns 403                                                                                    | Probe events on the pod |
| Service-mesh sidecar (Istio, Linkerd) probing the upstream cluster by IP returns 403                                                                       | Sidecar access log      |
| `curl -k https://1.2.3.4/` to a real IP behind a hosted LB returns the edge block page, not the app                                                        | Manual repro            |
| Long-running monitoring tool that pinned an old IP returns 403 after a DNS rotation                                                                        | Monitoring dashboard    |
| Vulnerability scanners (Qualys, Nessus, internal `nmap` scripts) reporting "site returned 403 on direct-IP probe"                                          | Scanner output          |

Distinguishing fingerprint: the failing requests carry a `Host:` header that is **parseable as an IP address** (with or without a port), not as a hostname. Browsers virtually never send Host as an IP; programmatic clients sometimes do (deliberately or accidentally).

## 2. The Diagnosis

### 2.1 What rule `920350` does

Rule `920350` is a regex rule against the `Host` header value. Its modsecurity-style signature:

```
SecRule REQUEST_HEADERS:Host "@rx ^(?:[\d.:]+|\[[\da-f:]+\])(?::\d+)?$"
  "id:920350,
   phase:1,
   block,
   capture,
   t:none,t:lowercase,
   msg:'Host header is a numeric IP address',
   tag:'attack-protocol',
   tag:'OWASP_CRS/PROTOCOL_VIOLATION/IP_HOST',
   ..."
```

Key facts:

- The regex matches:
  - Pure-IPv4: `203.0.113.5`, `10.0.0.42`
  - IPv4 with port: `203.0.113.5:443`, `127.0.0.1:8080`
  - Bracketed IPv6: `[2001:db8::1]`, `[::1]:443`
  - Borderline: any all-numeric-and-dots-and-colons string (catches some malformed but IP-shaped values too)
- Phase 1 (request headers). Terminates before body read.
- The rule does **not** distinguish private (`10.0.0.0/8`, `192.168.0.0/16`, `172.16.0.0/12`) from public IPs â€” both fire.
- It does **not** care whether the IP actually resolves to the served site. Even sending `Host: 1.1.1.1` to your own site fires the rule.

### 2.2 Why legitimate clients send IP in Host

| Client / scenario                                                                                                             | Why it sends IP                                               |
| ----------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------- |
| Kubernetes liveness / readiness probe with `httpGet.host` unset and default behavior using the pod IP                         | Default kubelet probe behavior â€” Host header is the pod's IP. |
| Service-mesh upstream probe (Envoy, Linkerd2 proxy) probing the upstream cluster member by IP                                 | The mesh dials by IP for load-balancing precision; Host = IP. |
| Long-poll / WebSocket clients that pinned an old IP after first DNS lookup                                                    | TTL expired but the client cached forever.                    |
| Vulnerability scanner doing `nmap --script http-â€¦` against a CIDR range                                                       | Scanner targets each IP; Host header is the IP under test.    |
| Internal cron job that talks to a service via IP because DNS isn't available in that runtime                                  | Stripped-down container with no `/etc/resolv.conf` config.    |
| Browser-as-developer hitting `https://10.0.0.42` directly during development                                                  | Manual debug â€” Host is the IP.                                |
| Reverse-proxy misconfiguration where `proxy_set_header Host $proxy_host;` is used and `$proxy_host` resolves to an IP literal | nginx config bug.                                             |
| HAProxy `backend` health check with `httpchk` default â€” sends `Host: ` to the backend IP                                      | HAProxy config left default.                                  |

### 2.3 The single most common FP cluster

**Kubernetes pod-IP probes**, especially when the WAF is deployed as a sidecar in front of every workload (rare but real in defense-in-depth setups) or when an HTTPS LB has both an external VIP and a private-IP path that the kubelet uses for liveness. Symptom: deploys succeed, then 60 seconds later the pod gets killed because liveness fails â€” and the failing requests all have `Host: <pod-ip>`.

### 2.4 Sibling rules that co-fire

| Rule                | What it checks        | Why it co-fires                                            |
| ------------------- | --------------------- | ---------------------------------------------------------- |
| `920350`            | Host is numeric IP    | The primary terminator.                                    |
| `920280`            | Missing Host          | Mutually exclusive (you have IP-as-Host vs no Host).       |
| `920310` / `920311` | Empty `Accept` header | Co-fires when the same minimalist probe also lacks Accept. |
| `920320`            | Missing User-Agent    | Co-fires on minimalist probes.                             |
| `920330`            | Empty User-Agent      | Same.                                                      |

Probe-shaped requests very commonly miss multiple HTTP/1.1 niceties at once; expect to tune multiple `920xxx` rules together for the same probe source.

## 4. The Remediation Matrix

> **Strong preference: fix the client.** A request with Host=IP is a real protocol smell that often points to a misconfigured probe, a missing DNS record, or a cached stale IP. The right answer in most cases is to configure the probe / mesh / kubelet to send the correct hostname. Use a WAF exclusion only when the client truly cannot send a hostname (vendor-locked scanner, kubelet probe pointing at pod IP).

### 4.1 GCP Cloud Armor

Allow rule scoped by source IP CIDR (the probe network) and exact path:

```hcl
rule {
  action   = "allow"
  priority = 110
  preview  = true

  match {
    expr {
      expression = <<-CEL
        request.path == '/healthz'
        && inIpRange(origin.ip, '10.0.0.0/8')
      CEL
    }
  }

  description = "Allow internal-network /healthz probes despite Host=podIP. Kubelet liveness sends Host=podIP by default; configuring httpGet.host requires kubelet manifest change tracked in JIRA-9101. EXCEPTIONS.md row YYYY-MM-DD."
}
```

### 4.2 AWS WAF

If using `AWSManagedRulesKnownBadInputsRuleSet`, override the specific Host-related rules to Count for the affected IP set + path:

```hcl
rule {
  name     = "AWS-AWSManagedRulesKnownBadInputsRuleSet"
  priority = 5

  override_action { none {} }

  statement {
    managed_rule_group_statement {
      vendor_name = "AWS"
      name        = "AWSManagedRulesKnownBadInputsRuleSet"

      rule_action_override {
        name = "Host_localhost_HEADER"
        action_to_use { count {} }
      }

      scope_down_statement {
        and_statement {
          statement {
            ip_set_reference_statement {
              arn = aws_wafv2_ip_set.internal_probes.arn
            }
          }
          statement {
            byte_match_statement {
              field_to_match { uri_path {} }
              positional_constraint = "EXACTLY"
              search_string         = "/healthz"
              text_transformation { priority = 0; type = "NONE" }
            }
          }
        }
      }
    }
  }

  visibility_config {
    cloudwatch_metrics_enabled = true
    metric_name                = "kbi-host-override"
    sampled_requests_enabled   = true
  }
}
```

For a generic "Host is IP" custom rule on AWS:

```hcl
rule {
  name     = "deny-host-is-ip-except-internal-healthz"
  priority = 200
  action { block {} }

  statement {
    and_statement {
      statement {
        regex_match_statement {
          regex_string = "^(?:[0-9]{1,3}\\.){3}[0-9]{1,3}(?::[0-9]+)?$"
          field_to_match { single_header { name = "host" } }
          text_transformation { priority = 0; type = "LOWERCASE" }
        }
      }
      statement {
        not_statement {
          statement {
            and_statement {
              statement {
                ip_set_reference_statement {
                  arn = aws_wafv2_ip_set.internal_probes.arn
                }
              }
              statement {
                byte_match_statement {
                  field_to_match { uri_path {} }
                  positional_constraint = "EXACTLY"
                  search_string         = "/healthz"
                  text_transformation { priority = 0; type = "NONE" }
                }
              }
            }
          }
        }
      }
    }
  }

  visibility_config {
    cloudwatch_metrics_enabled = true
    metric_name                = "deny-host-is-ip"
    sampled_requests_enabled   = true
  }
}
```

### 4.3 Cloudflare

```hcl
resource "cloudflare_ruleset" "skip_managed_for_internal_probes" {
  zone_id = var.zone_id
  name    = "Skip managed for internal probes"
  kind    = "zone"
  phase   = "http_request_firewall_custom"

  rules {
    action = "skip"
    action_parameters {
      ruleset = "current"
      phases  = ["http_request_firewall_managed"]
    }
    expression  = "(ip.src in {10.0.0.0/8} and http.request.uri.path eq \"/healthz\")"
    description = "Skip managed WAF for internal /healthz probes (Host=podIP). EXCEPTIONS.md row YYYY-MM-DD."
    enabled     = true
  }
}
```

> Cloudflare's edge enforces Host vs the zone's proxied hostnames at the connection layer. A direct `https://<cloudflare-edge-ip>/` request with `Host: 1.2.3.4` will be rejected before WAF rules run; WAF tuning does not help in that case.

### 4.4 Verification

- Internal probe stops returning 403; kubelet stops killing the pod.
- External request with Host=IP from a non-allowed IP range still returns 403.
- Re-running `curl -k --resolve â€¦` from a non-internal IP returns 403 (the rule still works).

## 6. Common pitfalls

- **Connection-layer Host validation.** Cloudflare and (to a lesser extent) AWS reject Host=IP at the connection / front-end layer before WAF rules run. If the rejection comes from there, WAF tuning has no effect. Identify the layer first.
- **`AWSManagedRulesKnownBadInputsRuleSet` is not a CRS port.** AWS lacks a direct `920350` equivalent. The closest match is `Host_localhost_HEADER`, which catches `Host: localhost` but not arbitrary IPs. For full coverage you need a custom rule.
- **IPv6 brackets.** Many clients send `Host: 2001:db8::1` (no brackets); spec requires `[2001:db8::1]`. Both should be denied â€” confirm both forms in your audit row.
- **IP-allow-listing internal CIDRs.** The allow rule should be scoped as tightly as possible (preferably to specific node IP ranges, not all RFC1918). Wide internal allows are a path to lateral-movement risk.
- **`X-Forwarded-Host` is not Host.** Some apps look at `X-Forwarded-Host` and treat it as authoritative. The rule `920350` only inspects the literal `Host` header. If your app trusts `X-Forwarded-Host`, that is a separate bug class (host-header injection).
- **Sticky kubelet behavior.** Setting `httpGet.host` in a Pod spec only changes the Host header for the probe; it does not change the IP the probe targets. The probe still hits the pod IP â€” but with `Host: <chosen-hostname>`. Confirm with `kubectl describe pod` that the override took effect.
- **Probe IP rotation.** Kubernetes assigns pod IPs from a range, and they rotate per deploy. Allow rules should scope by **CIDR range**, not individual IPs.
