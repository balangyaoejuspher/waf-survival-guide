# OWASP CRS Rule `913100` — Scanner / Security Tool Detection

> **Scope of this file:** the single canonical reference for OWASP CRS rule `913100` — User-Agent-based detection of known security scanners and pentest tools (Acunetix, Burp, Nessus, Nikto, OpenVAS, Qualys, sqlmap, w3af, Wapiti, ZAP, Metasploit, etc.) — and the false positives it produces on **authorized** internal pentests, CI security scans, bug-bounty researcher engagements, and legitimate tools that happen to ship with a default scanner-like User-Agent. Covers Google Cloud Armor, AWS WAF, and Cloudflare WAF side by side.

---

## 1. The Symptom

| Cross-provider signature | Where you see it |
| --- | --- |
| `HTTP 403` on every request from an **authorized** pentest / scanner run, even though the scope was pre-approved | Pentest team's tool output |
| CI / nightly DAST job (OWASP ZAP, Burp Enterprise, Acunetix container) returns 100% finding "site refuses requests" | CI logs |
| `curl -A "sqlmap/1.7.6"` from a legitimate dev machine investigating a real SQL-injection finding gets blocked | Manual repro |
| A SaaS security-monitoring service (e.g. a third-party "external attack surface" scanner) you contracted shows your site as "unscannable" | Vendor dashboard |
| Internal asset inventory tool using `nmap --script http-…` to enumerate hosts gets blocked | Tooling logs |
| Sibling rules `913110` / `913120` (security tool / scripting language detection) firing alongside | Logged rule-ID list |

Distinguishing fingerprint: the failing request set originates from a **known scanner tool**, and the User-Agent string contains the tool's signature substring. Removing or randomizing the User-Agent eliminates the block.

---

## 2. The Diagnosis

### 2.1 What rule `913100` does

Rule `913100` is a regex rule against the `User-Agent` request header. Its modsecurity-style signature:

```
SecRule REQUEST_HEADERS:User-Agent "@pmFromFile scanners-user-agents.data"
  "id:913100,
   phase:1,
   block,
   t:none,t:lowercase,
   msg:'Found User-Agent associated with security scanner',
   tag:'attack-reputation-scanner',
   ..."
```

Key facts:

- Match operator is `@pmFromFile` — a fast Aho-Corasick **substring** match against the contents of `scanners-user-agents.data`, a CRS-shipped file with ~250+ scanner User-Agent substrings.
- Examples of substrings the file contains: `sqlmap`, `nikto`, `acunetix`, `burp`, `w3af`, `wpscan`, `qualys`, `nessus`, `openvas`, `arachni`, `webinspect`, `wapiti`, `zap`, `dirbuster`, `gobuster`, `nuclei`, `nmap`, `masscan`, `python-requests` (in some CRS versions, controversially), `go-http-client` (also controversial).
- Phase 1 (request headers). Terminates before body read.
- The match is case-insensitive (`t:lowercase`).
- Substring matching means **any** occurrence triggers — a User-Agent like `Mozilla/5.0 (sqlmap-investigation-machine)` will trip on `sqlmap`.
- Companion rules in the `913xxx` family:
  - `913101` — alternative scanner UAs (separate data file)
  - `913110` — security-scanner specific UAs (Acunetix, etc., often duplicated)
  - `913120` — scripting-language default UAs (`python-urllib`, `Wget`, `curl`, `libwww-perl`, `Java/`) — **this one is the high-FP rule**, often disabled in production

### 2.2 The legitimate inputs that trip it

| Submitted shape | Why the rule fires |
| --- | --- |
| Internal pentest with `sqlmap -u "…"` | `User-Agent: sqlmap/1.x.y` matches `sqlmap`. |
| OWASP ZAP-based CI scan with default UA | `User-Agent: Mozilla/5.0 (Mozilla/5.0; ZAP/2.x.y)` matches `zap`. |
| Burp Suite scanner during a contracted engagement | UA contains `Burp` or sub-tool name. |
| Nuclei template-driven scan against your own staging environment | UA matches `nuclei`. |
| Internal asset scanner (`nmap --script http-title -p 443 …`) | UA matches `nmap`. |
| Researcher submitting a bug-bounty repro using their everyday `curl --user-agent "sqlmap test"` | Substring match. |
| Backend service that legitimately uses `python-requests` (matched by `913120` if enabled, not `913100`) | UA is the default Python requests library string. |
| Go-based internal API client with default `Go-http-client/1.1` UA (matched by `913120`, not `913100`) | Default Go HTTP client. |
| Synthetic check tool (`Pingdom`, `Datadog`) UA, where the vendor's UA happens to contain a substring matched by the data file | Rare but possible. |

### 2.3 The crucial point — `913100` is doing exactly what you want, **except** during authorized scans

`913100` is not a typical "false positive" rule in the same sense as XSS or SQLi rules. The Internet at scale **is** continuously being scanned by `sqlmap` / `nikto` / `nuclei` and similar. Blocking them at the edge is correct and desirable; it reduces log noise and slows opportunistic attackers.

The FP scenarios are narrow:

1. **Authorized internal pentest** — you want the scanner to see real responses to assess real risk.
2. **CI / nightly DAST scan** — your CI runs ZAP or similar against staging.
3. **Bug-bounty researcher** — pre-approved scope, they need to scan.
4. **Vendor security service** — contracted external attack-surface scanner.

In every case the fix is **source-IP allow-list**, not blanket UA tuning. Never disable `913100` globally — that re-floods your logs with opportunistic scanner traffic and removes a useful threat-intel signal.

### 2.4 Sibling rules to be aware of

| Rule | Detector | Notes |
| --- | --- | --- |
| `913100` | Scanner UA substring match | The primary terminator for tool-shaped UAs. |
| `913101` | Alternative scanner UA list | Independent file; some scanners are only in this list. |
| `913110` | Security-scanner specific UAs | Overlaps `913100`. |
| `913120` | Scripting-language default UAs (`python-urllib`, `Wget`, `Java/`, etc.) | **High FP rate.** Many legitimate microservices use these. Usually **disabled** in production via paranoia-level tuning or per-policy exclusion. |

### 2.5 Provider mapping

| Provider | Where `913100` lives | Notes |
| --- | --- | --- |
| **GCP Cloud Armor** | Preconfigured ruleset `scannerdetection-stable` (or `scannerdetection-v33-stable`), rule ID `owasp-crs-v030301-id913100-scannerdetection` | Literal CRS port. |
| **AWS WAF** | `AWSManagedRulesKnownBadInputsRuleSet` includes scanner-UA blocks, but the rule names are AWS-specific (e.g. `BadAuthToken`, `JavaDeserializationRCE`, `Host_localhost_HEADER`); there is no exact CRS `913100` equivalent. **AWS does not ship a dedicated managed scanner-UA rule group.** For comprehensive scanner-UA blocking on AWS, build a custom rule that matches User-Agent against a regex pattern or pulls from a maintained IP / UA list. | Different model. |
| **Cloudflare WAF** | "Cloudflare OWASP Core Ruleset" — `OWASP_CRS_V032_913100`. Cloudflare additionally has **"Threats" / "Security Level"** features that block known bad reputation IPs before WAF rules run, plus the **Cloudflare Managed Ruleset** with its own scanner-UA heuristics. | Literal CRS port. Double-coverage applies. |

## 3. The Log Evidence

### 3.1 GCP Cloud Armor

```text
resource.type="http_load_balancer"
AND jsonPayload.enforcedSecurityPolicy.outcome="DENY"
AND jsonPayload.enforcedSecurityPolicy.preconfiguredExprIds:"owasp-crs-v030301-id913100-scannerdetection"
AND timestamp >= timestamp_sub(@end, INTERVAL 1 HOUR)
```

The matched User-Agent string is logged in `httpRequest.userAgent` (Cloud Armor preserves UA), so confirming which scanner triggered is trivial:

```text
resource.type="http_load_balancer"
AND jsonPayload.enforcedSecurityPolicy.outcome="DENY"
AND jsonPayload.enforcedSecurityPolicy.preconfiguredExprIds:"owasp-crs-v030301-id913100-scannerdetection"
| stats count by httpRequest.userAgent, httpRequest.remoteIp
```

### 3.2 AWS WAF

If you've built a custom scanner-UA rule, query by its name. Otherwise, AWS does not provide an out-of-the-box query for this.

```text
fields @timestamp, action, terminatingRuleId, httpRequest.headers, httpRequest.clientIp
| filter action = "BLOCK"
| filter @message like /sqlmap/ or @message like /nikto/ or @message like /nuclei/ or @message like /zap/
| sort @timestamp desc
| limit 50
```

### 3.3 Cloudflare

```bash
curl -s "https://api.cloudflare.com/client/v4/zones/$ZONE_ID/security/events?datetime_geq=2026-06-17T10:00:00Z" \
  -H "Authorization: Bearer $CF_TOKEN" \
  | jq '.result[] | select(.rule_id | test("913100")) |
        {ray_id, rule_id, rule_message, action, client_ip, user_agent, occurred_at}'
```

### 3.4 Local reproduction

```bash
curl -v -A "sqlmap/1.7.6#stable (https://sqlmap.org)" \
     "https://app.example.com/" 2>&1 | tail -20
```

To verify which substring is matched, walk the data file:

```bash
curl -s https://raw.githubusercontent.com/coreruleset/coreruleset/main/rules/scanners-user-agents.data \
  | tee /tmp/scanners.txt | wc -l
```

Grep the file for substrings present in your failing UA:

```bash
grep -F -f <(echo "sqlmap"; echo "nikto"; echo "zap") /tmp/scanners.txt
```

### 3.5 Offline check

```bash
python - <<'PY'
import urllib.request

url = "https://raw.githubusercontent.com/coreruleset/coreruleset/main/rules/scanners-user-agents.data"
data = urllib.request.urlopen(url).read().decode().splitlines()
substrings = [s.lower().strip() for s in data if s.strip() and not s.startswith("#")]

ua = "Mozilla/5.0 (sqlmap-investigation-machine)"  # the UA under test
ua_lower = ua.lower()
matches = [s for s in substrings if s in ua_lower]
print(f"matched substrings: {matches}")
PY
```

## 4. The Remediation Matrix

> **House rule for `913100`:** never disable globally; tune by **source IP allow-list** scoped to the contracted scanner's known address range, and ideally combined with a **time-window** for the scheduled scan. Pair the WAF allow with a written engagement scope (ticket / contract / bug-bounty platform record) referenced from the audit row.

### 4.1 GCP Cloud Armor

Allow rule scoped by source IP (the pentest team's egress, the CI runner's IP, the contracted vendor's published scanner range):

```hcl
rule {
  action   = "allow"
  priority = 50
  preview  = true

  match {
    expr {
      expression = <<-CEL
        inIpRange(origin.ip, '198.51.100.0/29')   // bug-bounty researcher Alice, scope ticket SEC-2026-117
        || inIpRange(origin.ip, '203.0.113.10/32') // ZAP CI runner static egress IP
      CEL
    }
  }

  description = "Allow pre-approved scanner sources to bypass 913100. SEC-2026-117 (bounty Alice), CI ZAP runner. Time-bounded review +30d. EXCEPTIONS.md row YYYY-MM-DD."
}
```

The allow rule runs at priority 50 (high precedence). It bypasses **all** WAF rules for those IPs on every path — which is what an authorized scanner needs.

> **Tighten by path if possible.** If the contracted scope is `*.staging.example.com` only, scope the allow by `request.host.matches(...)`. A blanket "all paths, all rules" allow is risky if the IP range is later reassigned by the vendor.

### 4.2 AWS WAF

```hcl
resource "aws_wafv2_ip_set" "pre_approved_scanners" {
  name               = "pre-approved-scanners"
  scope              = "REGIONAL"
  ip_address_version = "IPV4"
  addresses = [
    "198.51.100.0/29",  # SEC-2026-117 bounty
    "203.0.113.10/32",  # CI ZAP runner
  ]
}

rule {
  name     = "allow-pre-approved-scanners"
  priority = 1
  action { allow {} }

  statement {
    ip_set_reference_statement {
      arn = aws_wafv2_ip_set.pre_approved_scanners.arn
    }
  }

  visibility_config {
    cloudwatch_metrics_enabled = true
    metric_name                = "allow-scanners"
    sampled_requests_enabled   = true
  }
}
```

Place this allow rule at the **lowest priority number** (highest precedence) in the Web ACL so it short-circuits all subsequent managed rules.

### 4.3 Cloudflare

Custom rule with `Skip` action targeting all managed phases:

```hcl
resource "cloudflare_ruleset" "skip_managed_for_approved_scanners" {
  zone_id = var.zone_id
  name    = "Skip managed for pre-approved scanners"
  kind    = "zone"
  phase   = "http_request_firewall_custom"

  rules {
    action = "skip"
    action_parameters {
      ruleset = "current"
      phases  = ["http_request_firewall_managed"]
      products = ["waf", "rateLimit", "securityLevel"]
    }
    expression  = "(ip.src in {198.51.100.0/29 203.0.113.10/32})"
    description = "Skip managed WAF + rate-limit + security-level for approved scanners. SEC-2026-117 (bounty), CI ZAP. Review +30d. EXCEPTIONS.md row YYYY-MM-DD."
    enabled     = true
  }
}
```

> Cloudflare's "Security Level" (separate from WAF rules) and "I'm Under Attack" mode can also block scanners independently. Adding `securityLevel` to the `products` list skips those too. Confirm the scanner can actually reach origin by re-running the scan after enabling.

### 4.4 Verification

- The contracted scanner's traffic reaches origin and produces findings (or 200 OK on normal pages, depending on what the scanner does).
- A non-allowed IP making the same `sqlmap` request still returns 403.
- App access log shows the scanner's requests with timestamps matching the scheduled window.

## 5. Audit Trail

```
| 2026-06-17 | staging.example.com | gcp-cloud-armor | 913100 (bypassed via priority-50 allow) | inIpRange(198.51.100.0/29, 203.0.113.10/32) | SEC-2026-117 (HackerOne bug-bounty engagement, Alice Researcher, scope = *.staging.example.com, window 2026-06-17→2026-07-17). CI ZAP scan from runner egress IP. | <link to HackerOne ticket + CI run> | preview 2026-06-17 → enforce 2026-06-18 | @secops-lead | 2026-07-17 (hard cutoff: scope expiry) |
```

For scanner allows, **set the Review by date equal to the engagement scope's expiry**, not the default +180. Hard cutoff. When the bounty / pentest ends, the allow rule must be removed within 24h.

## 6. Common pitfalls

- **The allow rule is broader than intended.** A `Skip` of all WAF phases for an IP means SQLi / XSS / RCE / rate-limit are all bypassed for that IP. This is correct for an authorized scanner but catastrophic if the IP is later reassigned to a different tenant by the vendor. Refresh the vendor IP list at engagement start.
- **Scanner IPs rotate.** Bug-bounty researchers often work from VPNs that rotate IPs daily. A static allow rule may stop working mid-engagement. Coordinate via the bug-bounty platform's documented researcher IP set, or have the researcher confirm a stable egress IP.
- **Don't allow by User-Agent alone.** An attacker can trivially spoof `User-Agent: sqlmap/...` to bypass a UA-keyed allow. Always combine UA-based allows with source-IP constraints.
- **`913120` (scripting-language default UAs) is a separate problem.** It catches `python-requests`, `Go-http-client`, `Wget`, `Java/`. Many legitimate backend services have these default UAs. Tune `913120` per service, not globally; or have services set a meaningful `User-Agent: <service-name>/<version>` so they don't get caught at all.
- **CI runners with shared egress.** A shared CI runner pool means a noisy CI job from a different team could trip 913100 from the same allow-listed IP and consume your debugging time. Use dedicated CI egress for security scans.
- **Time-bounded allows.** A scanner allow rule is not "set and forget". Set the Review by date to the engagement expiry, not the default +180.
- **Cloudflare "Security Level" interaction.** Cloudflare's reputation-based blocking runs before WAF; a researcher's IP may be on Cloudflare's threat list independent of your rule. Adding the IP to a Cloudflare IP Access Rule (whitelist) at the zone level may be necessary in addition to the WAF skip.
