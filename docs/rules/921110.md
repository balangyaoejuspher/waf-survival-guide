# OWASP CRS Rule `921110` — HTTP Request Smuggling

> **Scope of this file:** the single canonical reference for OWASP CRS rule `921110` — protocol-enforcement rule that denies requests showing classic HTTP request-smuggling signatures: simultaneous `Transfer-Encoding` and `Content-Length` headers, duplicated `Transfer-Encoding`, conflicting `TE`/`CL` values, embedded HTTP request lines inside header values — and the false positives it produces on legitimate proxies that _intentionally_ set both headers during chunked-to-fixed conversion, gRPC-Web gateways, some misbehaving CDN-to-origin paths, and Java/Tomcat HTTP clients with old `Transfer-Encoding` handling. Covers Google Cloud Armor, AWS WAF, and Cloudflare WAF side by side.
>
> Companion rules: `921120` (HTTP response splitting via CRLF in header values), `921130` (HTTP response splitting via CRLF in `Set-Cookie`-like sinks), `921150` / `921160` (additional smuggling variants).

---

## 1. The Symptom

| Cross-provider signature                                                                                                                                   | Where you see it                   |
| ---------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------- |
| `HTTP 403` (or 400 at front-end) on POST/PUT requests originating from a specific upstream proxy or service-mesh sidecar                                   | Edge log, correlated with proxy IP |
| Cluster of denies starting immediately after a proxy software upgrade (nginx, HAProxy, Envoy, Istio)                                                       | Deploy timeline                    |
| gRPC-Web translation gateway requests fail intermittently                                                                                                  | gRPC gateway logs                  |
| Webhook deliveries from a SaaS partner that uses a custom HTTP client (instead of standard libcurl) — partner's logs show "200 sent" but you never receive | Partner support thread             |
| Old Java application servers (Tomcat 7, JBoss EAP 6) acting as upstream of the WAF — `Transfer-Encoding: chunked` + `Content-Length` both present          | Upstream server logs               |
| Edge log shows the rule terminator as `921110` (or AWS `HostingProviderIPLeak` / Cloudflare equivalent)                                                    | Rule ID in log                     |

Distinguishing fingerprint: the failing request has headers that look "ambiguous about its own length". The request might be valid in some interpretations and invalid in others — which is exactly the smuggling condition the rule is designed to catch.

---

## 2. The Diagnosis

### 2.1 What rule `921110` does

Rule `921110` is one of several rules in the `921xxx` family that enforce strict HTTP-framing rules from RFC 7230 §3.3.3. The core signature shape:

```
SecRule REQUEST_HEADERS:Transfer-Encoding "@rx ^(.*)$"
  "id:921110,
   phase:1,
   block,
   chain,
   t:none,t:lowercase,
   msg:'HTTP Request Smuggling Attack',
   ..."
SecRule REQUEST_HEADERS:Content-Length "@rx ^(.*)$"
  "chain"
SecRule TX:0 "@streq chunked"
  "ctl:auditLogParts=+E"
```

(Reproduced for illustration — the exact CRS implementation evolves across versions; see the linked source.)

Key facts:

- The rule terminates when **both** `Transfer-Encoding` and `Content-Length` are present **and** `Transfer-Encoding` is `chunked`.
- RFC 7230 §3.3.3 mandates: if both are present, the request MUST be rejected (or `Content-Length` ignored). CRS chooses the strict "reject" interpretation.
- Sibling rules in the family catch:
  - Duplicated `Transfer-Encoding` (`Transfer-Encoding: chunked\r\nTransfer-Encoding: chunked`)
  - CR/LF embedded in header values (`921120` / `921130`)
  - `Transfer-Encoding: chunked, identity` (multiple TE values)
  - Smuggled HTTP request lines inside any header value
- Phase 1; terminates before body read.

### 2.2 Why this rule has fewer FPs than you'd expect — and why the FPs it does have are dangerous to ignore

`921110` catches a real, exploitable attack class (HTTP request smuggling — see [CVE-2019-18277](https://nvd.nist.gov/vuln/detail/CVE-2019-18277), Watchtowr / James Kettle's research). Disabling it removes a meaningful defense against very serious bugs. The legitimate cases that trip it are narrow:

| Submitted shape                                                                                                                  | Why it (legitimately) fires                        |
| -------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------- |
| A proxy that converts chunked → fixed-length but forgets to **remove** the `Transfer-Encoding: chunked` header before forwarding | nginx/HAProxy/Envoy bug — both headers present.    |
| A gRPC-Web → gRPC translation gateway forwarding the original `Transfer-Encoding` AND adding its own `Content-Length`            | Translation-layer header-set bug.                  |
| Two-layer proxy chain where each layer thought the other would strip the header                                                  | Architectural ambiguity; classic smuggling vector. |
| Java HTTP client `OkHttp` < 3.12, Jetty < 9.4.x, Tomcat 7 — known to send malformed framing on POST                              | Client-library bug; fixed in newer versions.       |
| Custom-written HTTP client that hand-rolls `Transfer-Encoding: chunked` and also helpfully computes `Content-Length`             | Author error.                                      |
| SaaS partner webhook using a custom HTTP client that has not been updated for RFC 7230 strictness                                | Partner-side bug.                                  |

**Every one of these is a real bug in the sending side.** The right fix is to fix the sender. The wrong fix is to disable `921110`. The only defensible "exclusion" is a tightly-scoped allow for a specific upstream IP range while you wait for the upstream fix to land, with a short Review-by date.

### 2.3 Sibling rules to be aware of

| Rule     | What it checks                                                                    | Notes                                         |
| -------- | --------------------------------------------------------------------------------- | --------------------------------------------- |
| `921110` | `TE: chunked` + `CL` both present                                                 | Primary terminator.                           |
| `921120` | CRLF in header values (response splitting / header injection prep)                | High-severity — almost never a legitimate FP. |
| `921130` | CRLF in cookie-like header sinks                                                  | Same.                                         |
| `921140` | Header value contains HTTP request line                                           | Catches embedded smuggled requests.           |
| `921150` | `Range` header with multi-segment numbers (CVE-2011-3192 Apache range header DoS) | Adjacent family.                              |
| `921160` | `TE: chunked, identity` (multi-value TE)                                          | Smuggling-adjacent.                           |

Co-firing is rare; each variant catches a distinct vector.

### 2.4 Provider mapping

| Provider            | Where `921110` lives                                                                                                                                                                                                                                                                                                            | Notes                                                                                                                                           |
| ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| **GCP Cloud Armor** | Preconfigured ruleset `protocolattack-stable`, rule ID `owasp-crs-v030301-id921110-protocolattack`                                                                                                                                                                                                                              | Literal CRS port. Cloud Armor + GCP HTTPS LB additionally enforce RFC 7230 framing at the front end; rejections may occur before WAF rules run. |
| **AWS WAF**         | `AWSManagedRulesKnownBadInputsRuleSet` includes some smuggling-related rules (e.g. headers with CRLF / NUL bytes); the framing-conflict case is enforced by ALB / CloudFront at the request-parsing layer **before** WAF runs. **AWS does not expose a dedicated smuggling-rule equivalent to `921110` in its managed groups.** | Architectural enforcement at ALB / CloudFront, not at WAF.                                                                                      |
| **Cloudflare WAF**  | "Cloudflare OWASP Core Ruleset" — `OWASP_CRS_V032_921110`. Cloudflare's front-end HTTP parser also enforces RFC 7230 strictly and may reject before WAF runs.                                                                                                                                                                   | Connection-layer enforcement is independent.                                                                                                    |

## 3. The Log Evidence

### 3.1 GCP Cloud Armor

```text
resource.type="http_load_balancer"
AND jsonPayload.enforcedSecurityPolicy.outcome="DENY"
AND jsonPayload.enforcedSecurityPolicy.preconfiguredExprIds:"owasp-crs-v030301-id921110-protocolattack"
AND timestamp >= timestamp_sub(@end, INTERVAL 1 HOUR)
```

Cluster by source IP / userAgent to identify which upstream is malformed:

```text
resource.type="http_load_balancer"
AND jsonPayload.enforcedSecurityPolicy.outcome="DENY"
AND jsonPayload.enforcedSecurityPolicy.preconfiguredExprIds:"owasp-crs-v030301-id921110-protocolattack"
| stats count by httpRequest.remoteIp, httpRequest.userAgent
```

If you see no `921110` matches but the symptom suggests smuggling, look for front-end-level rejections:

```text
resource.type="http_load_balancer"
AND httpRequest.status=400
AND jsonPayload.statusDetails=~"(?i)(transfer.?encoding|content.?length|chunk)"
```

### 3.2 AWS WAF / ALB

ALB access log (Athena):

```sql
SELECT request_url, elb_status_code, target_status_code, error_reason, request_processing_time, client_ip, user_agent
FROM alb_access_logs
WHERE elb_status_code = 400
  AND error_reason IN ('HttpInvalidHeader', 'HttpInvalidVersion', 'HttpInvalidContentLengthHeader')
  AND from_iso8601_timestamp(time) > now() - interval '1' hour
ORDER BY time DESC
LIMIT 50;
```

### 3.3 Cloudflare

```bash
curl -s "https://api.cloudflare.com/client/v4/zones/$ZONE_ID/security/events?datetime_geq=2026-06-17T10:00:00Z" \
  -H "Authorization: Bearer $CF_TOKEN" \
  | jq '.result[] | select(.rule_id | test("921110")) |
        {ray_id, rule_id, action, client_ip, user_agent, occurred_at}'
```

Front-end-level smuggling rejections (returned 400 by Cloudflare's parser, not WAF) do not appear in Security Events; they appear in Logpush `http_requests` with `ClientRequestStatus = 400` and a parser-error subfield.

### 3.4 Local reproduction

Constructing a smuggling-shaped request manually (for debugging your own traffic only — do not run against a system you do not own):

```bash
printf 'POST /webhook HTTP/1.1\r\nHost: app.example.com\r\nContent-Length: 5\r\nTransfer-Encoding: chunked\r\n\r\n0\r\n\r\nGET /admin HTTP/1.1\r\nHost: app.example.com\r\n\r\n' \
  | openssl s_client -connect app.example.com:443 -quiet 2>/dev/null | head -20
```

Expected response: `400 Bad Request` from the front end, or `403` from WAF if it reached `921110`.

### 3.5 Offline check

Inspect a captured request (e.g. `tcpdump` PCAP of the failing upstream):

- Are both `Transfer-Encoding` and `Content-Length` present?
- Are there multiple `Transfer-Encoding` headers?
- Is `Transfer-Encoding` something other than `chunked` alone (e.g. `chunked, identity`)?
- Are there CR/LF byte sequences in any header value other than the trailing `\r\n`?

Any "yes" confirms the rule is correctly firing on the upstream's malformed output.

## 4. The Remediation Matrix

> **Strong preference: fix the upstream sender.** Smuggling-shaped requests are almost always a real bug — usually in a proxy or HTTP client library. Identify the sender, file an upstream bug, and pin the library version once fixed.
>
> Use a WAF allow only as a **time-bounded** workaround while the upstream fix is in flight, scoped tightly by source IP **and** the specific affected route, with a Review by date measured in weeks, not months.

### 4.1 GCP Cloud Armor

Allow rule scoped by source IP CIDR and path:

```hcl
rule {
  action   = "allow"
  priority = 80
  preview  = true

  match {
    expr {
      expression = <<-CEL
        request.path == '/webhook/partner-x'
        && inIpRange(origin.ip, '198.51.100.32/27')
      CEL
    }
  }

  description = "Allow partner-x webhook IPs through 921110 (their HTTP client sends TE+CL; vendor patch ETA 2026-07-01). Webhook is HMAC-verified and idempotent. EXCEPTIONS.md row YYYY-MM-DD."
}
```

> **The allow bypasses ALL WAF rules for that path+IP combination.** That is a wide opening. Compensate at the app layer: HMAC-verify every webhook payload before processing, and ensure the webhook handler is idempotent (a smuggled secondary request appended to a webhook body would be visible as a second event in your idempotency layer).

### 4.2 AWS WAF / ALB

If rejection is at ALB (the common case), no WAF rule will help. Options:

- File an upstream bug; pin the partner-vendor's fix ETA.
- Route partner-x's webhooks through a different ingress (NLB → backend directly, bypassing ALB+WAF) — accept the documented coverage gap.
- Have the partner switch their HTTP client to a standard, RFC-compliant library.

If rejection is at WAF (rare), allow rule by IP set + path:

```hcl
rule {
  name     = "allow-partner-x-webhook"
  priority = 1
  action { allow {} }

  statement {
    and_statement {
      statement {
        ip_set_reference_statement {
          arn = aws_wafv2_ip_set.partner_x_webhook_ips.arn
        }
      }
      statement {
        byte_match_statement {
          field_to_match { uri_path {} }
          positional_constraint = "EXACTLY"
          search_string         = "/webhook/partner-x"
          text_transformation { priority = 0; type = "NONE" }
        }
      }
    }
  }

  visibility_config {
    cloudwatch_metrics_enabled = true
    metric_name                = "allow-partner-x"
    sampled_requests_enabled   = true
  }
}
```

### 4.3 Cloudflare

```hcl
resource "cloudflare_ruleset" "skip_managed_for_partner_x" {
  zone_id = var.zone_id
  name    = "Skip managed for partner-x webhook"
  kind    = "zone"
  phase   = "http_request_firewall_custom"

  rules {
    action = "skip"
    action_parameters {
      ruleset = "current"
      phases  = ["http_request_firewall_managed"]
    }
    expression  = "(ip.src in {198.51.100.32/27} and http.request.uri.path eq \"/webhook/partner-x\")"
    description = "Skip managed WAF for partner-x webhook (TE+CL framing). Partner fix ETA 2026-07-01. HMAC-verified at app layer. EXCEPTIONS.md row YYYY-MM-DD."
    enabled     = true
  }
}
```

> Cloudflare's connection-layer HTTP parser is strict. If your partner's smuggling-shaped request is being rejected at that layer (before WAF), the only solution is the partner fixing their client or routing partner traffic through a non-Cloudflare path.

### 4.4 Verification

- Partner-x webhook deliveries succeed; your idempotency layer shows one event per webhook (no smuggled-second-request appearing as a duplicate or unexpected event).
- A smuggling-shaped request from a non-allowed IP still returns 403/400.
- Post-fix: when partner confirms the upstream fix is deployed, remove the allow rule **the same day** and verify the partner traffic still flows.

## 5. Audit Trail

```
| 2026-06-17 | api.example.com | gcp-cloud-armor | 921110 (bypassed via priority-80 allow) | request.path == /webhook/partner-x from 198.51.100.32/27 | Partner-x HTTP client sends both TE: chunked and Content-Length on every webhook; vendor patch ETA 2026-07-01. Webhook handler HMAC-verifies signature (commit a1b2c3d) and is idempotent (idempotency-key header, commit d4e5f6g). Risk acknowledged: smuggling vector on this endpoint until partner patch lands. | <link to vendor support ticket + log query> | preview 2026-06-17 → enforce 2026-06-18 | @secops-lead + @app-lead | 2026-07-08 (hard cutoff: vendor ETA + 1 week buffer) |
```

This is the rule type where the audit row matters most. The exclusion explicitly opens a smuggling vector; the row must record:

- Why it's tolerated (vendor bug, no workaround).
- What compensating controls are in place (HMAC verification, idempotency).
- When it expires (vendor fix ETA + buffer; **not** a 180-day default).
- Two approvers (secops + the application owner), not one.

---

## 6. Common pitfalls

- **Front-end vs WAF enforcement.** GCP / AWS / Cloudflare front ends enforce RFC 7230 framing independently of WAF rules. A WAF allow does not bypass front-end rejection. Identify which layer is denying using local repro.
- **The "fix" of stripping `Content-Length` at your edge proxy is a smuggling vector by itself.** If you add an nginx rewrite that drops `Content-Length` when `Transfer-Encoding` is present, you've built the exact bug `921110` exists to prevent. Do not do this. Fix the sender, or accept the rejection.
- **The allow bypasses everything else too.** If you allow partner-x's IPs to skip the WAF, they also skip SQLi / XSS / RCE / rate-limit rules. Compensate with HMAC verification, idempotency, and aggressive payload validation at the app layer.
- **Cloudflare connection-layer rejection is opaque.** If a smuggling-shaped request is rejected by Cloudflare's HTTP parser before WAF, you'll see a 400 but no Security Events row. Use Logpush `http_requests` dataset with `ClientRequestStatus = 400` to find these.
- **CR/LF in header values is a different conversation.** `921120` / `921130` catch response-splitting prep. These are essentially never legitimate. If you find yourself wanting to exclude `921120`, you are about to enable a real exploit — get a second opinion before merging.
- **Hard expiry date in the audit row.** Smuggling exclusions are time-bounded. Build a calendar reminder + an alert (e.g. a CI job that fails if the audit row's Review-by date has passed and the row is not struck through) to force the conversation at the cutoff date.
