# OWASP CRS Rule `949110` — Inbound Anomaly Score Exceeded

> **Scope of this file:** the single canonical reference for OWASP CRS rule `949110` — the **cumulative-scoring catch-all** that aggregates every other inbound rule's anomaly score and denies the request when the total crosses the configured threshold — and the false positives it produces from "death by a thousand cuts": no single rule matches strongly enough to block, but many low-severity matches accumulate past the threshold. Covers Google Cloud Armor, AWS WAF, and Cloudflare WAF side by side.
>
> This rule is **structurally different** from every other rule in the CRS. Read §2 carefully before tuning.

---

## 1. The Symptom

| Cross-provider signature                                                                                                                                                                                                             | Where you see it                     |
| ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------ |
| `HTTP 403` whose log row shows **`949110` as the terminator** but lists 4–10 other rules in `preconfiguredExprIds` (e.g. `942130`, `941340`, `913120`, `920420`, `980130`) — none individually blocked, but their combined score did | Edge log                             |
| Requests succeed when sent with simpler payloads, fail when 3+ different "small" things land in the same request (long User-Agent + cookie with special chars + query string with `<` somewhere + Content-Type with parameters)      | Manual repro by trimming the request |
| FPs increase after a paranoia-level bump or a CRS package upgrade                                                                                                                                                                    | Deploy timeline                      |
| FPs disappear when individual contributing rules are excluded one at a time, but no single exclusion fixes everything                                                                                                                | Iterative tuning experience          |
| GraphQL / complex-JSON / heavy-form-data endpoints (where every field is individually low-risk but the aggregate is many fields)                                                                                                     | API gateway access log               |

Distinguishing fingerprint: the terminator is `949110` (or its Anomaly Mode equivalent on the provider), and the request payload is **complex** rather than visibly attack-shaped. A single SQLi attempt blocks at `942100`; a long, structurally weird-but-benign request blocks at `949110`.

## 2. The Diagnosis

### 2.1 What `949110` does — the anomaly-scoring model

OWASP CRS supports two modes:

1. **Self-Contained Mode** (`SecRuleEngine On`, `tx.blocking_paranoia_level = 0`): each rule with a positive score blocks immediately on match.
2. **Anomaly Scoring Mode** (the default since CRS 3.x): each rule **adds to a running score** instead of blocking; a separate "blocking" rule (`949110` for inbound, `980130` for outbound) evaluates the score at the end of the phase and blocks if the total exceeds the threshold.

Each rule has a severity tag (CRITICAL, ERROR, WARNING, NOTICE) which maps to a default score:

| Severity | Default score |
| -------- | ------------- |
| CRITICAL | 5             |
| ERROR    | 4             |
| WARNING  | 3             |
| NOTICE   | 2             |

The default inbound threshold is **5** (single CRITICAL or a few WARNINGs is enough to block). The `949110` rule definition (simplified):

```
SecRule TX:INBOUND_ANOMALY_SCORE "@ge %{tx.inbound_anomaly_score_threshold}"
  "id:949110,
   phase:2,
   block,
   t:none,
   msg:'Inbound Anomaly Score Exceeded (Total Score: %{TX.INBOUND_ANOMALY_SCORE})',
   ..."
```

Key facts:

- `949110` itself has no payload pattern. It only blocks based on the cumulative inbound score.
- The default threshold of 5 is aggressive. Many CRS deployments raise it to 7 or 10 in production.
- A single rule whose score is ≥ threshold blocks alone via `949110`.
- N rules at WARNING (3) each contribute 3 to the score. Two WARNINGs (6 total) > threshold (5) → block.
- The list of "small things" that contributed is in the log row as the full `preconfiguredExprIds` array.

### 2.2 Why the FPs feel mysterious

Three structural properties make `949110` FPs harder to diagnose than per-rule FPs:

1. **No single rule is "wrong".** Every contributing rule matched something. Disabling one doesn't necessarily fix the FP — another low-score match may still push the total past threshold.
2. **The cause is the request's overall shape, not any one element.** A request with a long User-Agent (1 point from `913120`), special chars in a cookie (3 points from `920170`), an HTML-looking string in a query param (3 points from `941100`), and a JSON body with a backtick (3 points from `932130`) reaches 10 — well past threshold — and blocks.
3. **The "fix" is multi-rule, not single-rule.** You typically need to exclude 2–4 rules together, or raise the threshold for the specific path. Both approaches widen the security envelope; the audit row gets noisier.

### 2.3 The typical FP shape

| Request property                                              | Common contributing rule                                    | Score contribution |
| ------------------------------------------------------------- | ----------------------------------------------------------- | ------------------ |
| Long, machine-generated User-Agent                            | `913120` (scripting-language UA), `920420` (content-type)   | 3 + 2              |
| Cookie value with `;`, `=`, `:`, base64 padding               | `920170` (special chars), `942130` (boolean SQLi by chance) | 3 + 3              |
| Query string with a `<` or `>` (legit XML / Markdown content) | `941100` (XSS)                                              | 5                  |
| JSON body field with a backtick (e.g. code snippet content)   | `932130` (shell metacharacters)                             | 5                  |
| Long URL path with multiple slashes                           | `920430` (HTTP version), `920272` (restricted-characters)   | 2 + 3              |
| Total — none individually blocking, but aggregate > threshold | `949110`                                                    | terminator         |

### 2.4 Provider mapping — the "anomaly mode" picture

| Provider            | Where `949110` lives                                                                                                                                                                                                                                                                                                   | Notes                                                                                                                                                                                                                                                                                                                         |
| ------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **GCP Cloud Armor** | Preconfigured ruleset `inbound-anomaly-stable`, rule ID `owasp-crs-v030301-id949110-inbound-anomaly`                                                                                                                                                                                                                   | Literal CRS port. Cloud Armor exposes the anomaly threshold via the `sensitivity` parameter (1–4) on preconfigured WAF expressions: sensitivity 1 ≈ threshold 5 (default CRS), sensitivity 2 ≈ threshold 7, etc. **Tuning the sensitivity is a global threshold change for the entire preconfigured ruleset on that policy.** |
| **AWS WAF**         | The managed rule groups (`Common`, `SQLi`, `KnownBadInputs`, etc.) do **not** use CRS-style anomaly scoring — each AWS managed rule blocks independently. | **Cloudflare WAF**  | "Cloudflare OWASP Core Ruleset" includes the CRS anomaly scoring. The ruleset's overall **paranoia level** (PL1–PL4) and **anomaly score threshold** are exposed as managed-ruleset settings (per zone). `OWASP_CRS_V032_949110` is the rule ID that records the block.                                                | Same threshold-tuning model as Cloud Armor.                                                                                                                                                                                                                                                                                   |

## 3. The Log Evidence

### 3.1 GCP Cloud Armor

Baseline:

```text
resource.type="http_load_balancer"
AND jsonPayload.enforcedSecurityPolicy.outcome="DENY"
AND jsonPayload.enforcedSecurityPolicy.preconfiguredExprIds:"owasp-crs-v030301-id949110-inbound-anomaly"
AND timestamp >= timestamp_sub(@end, INTERVAL 1 HOUR)
```

The single most useful query: enumerate which contributing rules co-fire with `949110`, sorted by frequency:

```text
resource.type="http_load_balancer"
AND jsonPayload.enforcedSecurityPolicy.outcome="DENY"
AND jsonPayload.enforcedSecurityPolicy.preconfiguredExprIds:"owasp-crs-v030301-id949110-inbound-anomaly"
| stats count() by jsonPayload.enforcedSecurityPolicy.preconfiguredExprIds
```

This produces a list like:

```
[949110, 941100, 920170, 913120, 932130]  -> 1,243 hits
[949110, 941100, 920170, 932130]          ->   876 hits
[949110, 920170, 942130, 913120]          ->   412 hits
...
```

The top combinations tell you which 3–4 contributing rules are responsible for the bulk of FPs. Tune those, not `949110` itself.

### 3.2 AWS WAF

Not applicable — AWS does not use anomaly scoring in its managed rule groups. If you are seeing "death by a thousand cuts"-shaped FPs on AWS, you are likely seeing individual rule blocks at high frequency, not score aggregation.

### 3.3 Cloudflare

```bash
curl -s "https://api.cloudflare.com/client/v4/zones/$ZONE_ID/security/events?datetime_geq=2026-06-17T10:00:00Z" \
  -H "Authorization: Bearer $CF_TOKEN" \
  | jq '.result[] | select(.rule_id | test("949110")) |
        {ray_id, rule_id, rule_message, action, uri, occurred_at, score: .matched_data.score}'
```

Cloudflare's event records include the computed anomaly score when payload logging is enabled.

### 3.4 Local reproduction

Trip multiple low-score rules in one request:

```bash
curl -v -A "python-requests/2.31.0 (custom-internal-client)" \
     -H "Cookie: SESSIONID=eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJ4IjoxfQ.x; debug=1" \
     -H "Content-Type: application/json; charset=utf-8" \
     -X POST "https://app.example.com/api/v1/notes?q=<unfinished" \
     --data '{"text":"see `man curl` for usage", "tags":["a","b","c"]}' 2>&1 | tail -30
```

This single request typically scores 8–12 against default CRS depending on policy, well past the default threshold of 5.

### 3.5 Offline check

Tally the scores for a target request manually:

```bash
python - <<'PY'
# Approximate; consult the actual rule files for exact severities.
contributing = [
    ("913120", "scripting UA", 2),
    ("920170", "special chars in cookie", 3),
    ("941100", "XSS suspect in q=", 5),
    ("932130", "shell metachar (backtick)", 5),
]
total = sum(s for _, _, s in contributing)
print(f"Total inbound score: {total}")
for rid, desc, s in contributing:
    print(f"  {rid}  +{s}  {desc}")
print(f"Threshold (default): 5 → block." if total >= 5 else "Below threshold.")
PY
```

## 4. The Remediation Matrix

> **House rule for `949110`:** never exclude `949110` itself. The rule is the threshold check; excluding it disables the whole anomaly-scoring model. Tune the **contributing rules** instead (with field/path scoping like other rules), OR raise the **threshold** for the specific path (the "less safe" option). Always prefer the former.
>
> Pair the tuning with a §3.1 frequency query before AND after, so the audit row demonstrates the FP cluster actually shrank.

### 4.1 GCP Cloud Armor — Option A: tune the contributing rules

Identify the top 3–4 contributing rules from §3.1, then build one exclusion rule per (path, contributing rule) combination, following the patterns in [docs/rules/941100.md §4.1](941100.md#41-gcp-cloud-armor), [docs/rules/932100.md §4.1](932100.md#41-gcp-cloud-armor), etc.

Example for the §3.4 reproduction request:

```hcl
rule {
  action   = "allow"
  priority = 930
  preview  = true

  match {
    expr {
      expression = "request.path.startsWith('/api/v1/notes')"
    }
  }

  preconfigured_waf_config {
    # Multiple exclusions of low-severity rules contributing to anomaly score.
    exclusion {
      target_rule_set = "scannerdetection-stable"
      target_rule_ids = ["owasp-crs-v030301-id913120-scannerdetection"]
      request_header {
        operator = "EQUALS"
        value    = "user-agent"
      }
    }
    exclusion {
      target_rule_set = "protocolattack-stable"
      target_rule_ids = ["owasp-crs-v030301-id920170-protocolattack"]
      request_cookie {
        operator = "EQUALS"
        value    = "SESSIONID"
      }
    }
    exclusion {
      target_rule_set = "rce-stable"
      target_rule_ids = ["owasp-crs-v030301-id932130-rce"]
      request_query_param {
        operator = "EQUALS"
        value    = "text"
      }
    }
  }

  description = "Reduce 949110 anomaly score on /api/v1/notes by excluding top 3 contributing rules: 913120 (UA), 920170 (cookie chars), 932130 (backticks in note text). Keeps SQLi/XSS/RCE rules active on other fields. See §3.1 query for frequency baseline. EXCEPTIONS.md row YYYY-MM-DD."
}
```

### 4.1b GCP Cloud Armor — Option B: raise the sensitivity

If contributing-rule tuning is impractical (too many distinct combinations), raise the path-scoped `sensitivity` parameter on `evaluatePreconfiguredWaf()`:

```hcl
rule {
  action   = "allow"
  priority = 935
  preview  = true
  match {
    expr {
      expression = <<-CEL
        request.path.startsWith('/api/v1/graphql')
        && evaluatePreconfiguredWaf('inbound-anomaly-stable', { 'sensitivity': 2 })
      CEL
    }
  }
  description = "Raise inbound anomaly sensitivity from 1 (threshold 5) to 2 (threshold 7) on /api/v1/graphql. GraphQL requests legitimately accumulate small per-field scores. Compensating control: field-level input validation in resolver (commit a1b2c3d). EXCEPTIONS.md row YYYY-MM-DD."
}
```

This is a **wider security envelope** than per-rule exclusion. Use only when per-rule tuning is impractical, and require a compensating-control reference in the audit row.

### 4.2 AWS WAF

Not applicable — AWS does not implement anomaly scoring in managed rules. "Death by a thousand cuts" symptoms on AWS are usually high-frequency individual rule blocks; debug them per-rule.

### 4.3 Cloudflare

Cloudflare exposes two tuning levers in the OWASP Core Ruleset:

- **Paranoia Level** (PL1–PL4) — enables / disables tranches of rules. PL2+ enables many additional rules that bump scores.
- **Score Threshold** — the cutoff at which `949110` blocks.

Console: **Security → WAF → Managed rules → Cloudflare OWASP Core Ruleset → Configure** → set **Paranoia Level** and **Score Threshold**.

Terraform (per-zone setting):

```hcl
resource "cloudflare_ruleset" "owasp_settings" {
  zone_id = var.zone_id
  name    = "OWASP Core Ruleset config"
  kind    = "zone"
  phase   = "http_request_firewall_managed"

  rules {
    action = "execute"
    action_parameters {
      id = "4814384a9e5d4991b9815dcfc25d2f1f"  # OWASP Core Ruleset

      overrides {
        rules {
          id     = "<rule-id-for-949110>"
          action = "block"        # keep blocking on threshold breach
          status = "enabled"
        }
      }
    }

    expression  = "true"
    description = "OWASP CRS active on all requests."
    enabled     = true
  }
}
```

Per-path threshold raise via a custom rule that skips the OWASP ruleset for a narrowly-scoped expression:

```hcl
resource "cloudflare_ruleset" "skip_owasp_for_graphql" {
  zone_id = var.zone_id
  name    = "Skip OWASP for GraphQL"
  kind    = "zone"
  phase   = "http_request_firewall_custom"

  rules {
    action = "skip"
    action_parameters {
      ruleset = "current"
      phases  = ["http_request_firewall_managed"]
      rulesets = ["4814384a9e5d4991b9815dcfc25d2f1f"]   # only the OWASP ruleset; Cloudflare Managed Ruleset still applies
    }
    expression  = "(http.request.uri.path eq \"/api/v1/graphql\" and http.request.method eq \"POST\")"
    description = "Skip OWASP CRS for /api/v1/graphql (accumulates anomaly score on legit complex queries). Cloudflare Managed Ruleset and rate-limit still apply. EXCEPTIONS.md row YYYY-MM-DD."
    enabled     = true
  }
}
```

### 4.4 Verification

Re-run the §3.1 frequency query for the affected path before AND after the exclusion:

- The combinations terminating on `949110` should drop dramatically (ideally to zero) for the tuned path.
- Other paths should be unaffected.
- A genuine SQLi attempt on the tuned path should still be blocked by `942100` directly (the per-rule blocks still fire; only the anomaly aggregation was tuned).

## 6. Common pitfalls

- **Never exclude `949110` directly.** It is the threshold check. Excluding it disables anomaly scoring entirely — every CRS rule reverts to immediate-block mode, which produces a tidal wave of FPs at the per-rule level. Always tune contributors, never the aggregator.
- **Frequency analysis BEFORE tuning is non-optional.** The §3.1 query identifies the actual top contributors. Tuning rules that "feel" responsible without confirming via data leads to several iterations of "still blocking", each requiring a new audit row.
- **Per-rule exclusions and threshold raises have very different audit weight.** A per-rule exclusion on a narrow field is the standard pattern (single approver, default review). A threshold raise on a path widens the envelope across many attack classes (two approvers, compensating-control reference required, shorter review window).
- **AWS doesn't have this problem.** If a team migrates from Cloudflare/Cloud Armor to AWS, "death by a thousand cuts"-shaped 403s disappear — and individual rule FPs become more visible instead. Tuning playbooks differ.
- **Paranoia level interactions (Cloudflare).** Raising paranoia from PL1 to PL2 enables additional contributing rules. A previously-tuned path can start hitting `949110` again after a paranoia bump even though no individual rule was changed.
- **Sensitivity vs threshold (Cloud Armor).** Cloud Armor's `sensitivity` parameter is just a friendlier name for the threshold. Sensitivity 1 = threshold 5, sensitivity 2 = threshold 7, etc. Don't get confused by the dual vocabulary.
- **Per-path threshold raises are a leak.** The `Skip` / sensitivity-bump approach is wider than per-rule tuning. Track skip-rule sprawl: a quarterly audit of all "skip OWASP for X" rules will catch the team's natural drift toward "just exclude more".
- **Wide-shape requests are sometimes a sign of bad client design.** A request that legitimately trips 4 different rule categories may indicate the client is putting too much in one request — paginate, split, or restructure rather than tune.
